# SmartRoute — Product Requirements Document (PRD)

> 서울 실시간 도시데이터를 활용해 **목적지까지의 실시간 교통**과 **주변 주차 가용 현황**을 한 번에 안내하는 멀티에이전트 챗 서비스.
> SDK: Microsoft Agent Framework (Python). 데이터: 서울 실시간 도시데이터(citydata, OA-21285).

---

## 1. 개요 / 문제정의
도심 운전자는 "어떻게 가야 덜 막히나"와 "도착지에 댈 곳이 있나"를 서로 다른 앱으로 확인한다.
SmartRoute는 이 둘을 **하나의 대화**로 통합하여, **실시간 데이터 기반으로 주차·교통을 동시 최적화**한 추천을 제공한다.

---

## 2. AI 에이전트 페르소나 (Personas)

### 2.1 Coordinator — "스마트루트 컨시어지"
- **성격/톤**: 침착하고 신뢰감 있는 길 안내 컨시어지. 군더더기 없이 핵심을 먼저 말한다.
- **역할**: 사용자 의도 파악, 목적지→핫스팟 매핑, 두 전문가 호출·종합, 최종 추천 제시.
- **말투 예시**: "강남역은 지금 진입로가 서행이에요. 대신 ○○주차장이 12면 여유라 그쪽으로 안내드릴게요."
- **원칙**: 수치는 전문가(도구) 결과만 인용, 트레이드오프는 솔직하게 설명.

### 2.2 ParkingAgent — "주차 도우미"
- **성격/톤**: 꼼꼼하고 실용적. "어디에, 몇 자리, 얼마"를 정확히 짚는다.
- **역할**: 지역 주차장별 총면/현재/여유면/요금/갱신시각을 분석해 **여유 우선**으로 추천.
- **원칙**: 만차/데이터 없음을 숨기지 않음. 대체 후보 제시.

### 2.3 RouteAgent — "교통 분석가"
- **성격/톤**: 데이터 지향적이고 간결. 혼잡 원인보다 "지금 어디가 덜 막히나"에 집중.
- **역할**: 지역 소통지수·평균속도·구간별 상태로 **접근 난이도/유리한 방향**을 설명.
- **원칙**: citydata 한계 내에서 "상대적으로 원활"까지만 안내(정밀 경로 단정 금지).

### 2.4 (참고) 사용자 페르소나
- **출퇴근 운전자**: 매일 같은 지역, "오늘 막히나/댈 곳 있나"를 빠르게.
- **방문 운전자**: 낯선 지역, "어디 대고 어떻게 접근"이 필요.

## 3. 오케스트레이션 방식

### 3.1 패턴: Coordinator + 병렬 전문가 (Concurrent → Synthesis)
```
        사용자 질의(목적지/의도)
                │
        ┌───────▼────────┐
        │  Coordinator   │  ① 의도 파악 ② 핫스팟 매핑
        └───────┬────────┘
        ③ 병렬 fan-out (동시 호출)
        ┌───────┴───────────────┐
        ▼                       ▼
  [RouteAgent]            [ParkingAgent]
  get_road_traffic        get_parking_status
        └───────┬───────────────┘
        ④ fan-in
        ┌───────▼────────┐
        │  Coordinator   │  ⑤ 결과 종합 → 단일 추천
        └───────┬────────┘
                ▼
        "○○ 접근 서행(18km/h) + △△주차장 12면 여유"
```

- **구현 방식(MAF Python)**: 두 전문가를 **agent-as-tool**로 Coordinator에 등록(`as_tool()` 류) → Coordinator LLM이 두 도구를 호출하고 종합. 동시 호출로 지연 최소화.
- **대안**: `agent_framework` 의 Workflow(Concurrent/Handoff)로 명시적 그래프 구성. 1차는 단순·견고한 agent-as-tool, 시간 여유 시 Workflow로 고도화.
- **상태 공유**: 핫스팟(확정 지역명)을 두 전문가에 동일 입력으로 전달. 전문가는 무상태(stateless)에 가깝게 유지.

### 3.2 오케스트레이션 가드 (요약, 상세는 AGENT.md)
- Coordinator만 사용자와 대화. 전문가는 확정된 지역명만 입력받음.
- 전문가 원시 수치는 재창작 없이 인용. 상충 결과는 트레이드오프로 설명.

## 4. 핵심 비즈니스 성공 지표 (KPIs)
> **북극성 지표(North Star): "실시간 데이터 활용을 통한 주차/교통 최적화로 절감된 사용자 총 소요시간"**

| 분류 | 지표 | 정의 / 목표 |
|------|------|-------------|
| **최적화 효과(핵심)** | 주차 탐색 시간 절감 | 추천 주차장 이용 시 평균 주차 탐색시간 ↓ (목표: 체감 30%↓) |
| | 접근 소요시간 절감 | 덜 막히는 접근 안내로 평균 접근시간 ↓ |
| | 추천 채택률 | 사용자가 추천 주차장/접근 방향을 실제 채택한 비율 (목표 ≥ 60%) |
| **데이터 신뢰성** | 데이터 신선도 | 응답에 사용된 citydata 갱신시각과 현재 시각 차 (목표 ≤ 5분) |
| | 실데이터 응답률 | 추측 없이 실제 도구 데이터로 답한 비율 (목표 ≥ 95%) |
| **품질** | 핫스팟 매핑 정확도 | 사용자 목적지→올바른 핫스팟 매핑 비율 |
| | 환각률(Hallucination) | 도구 미근거 수치 생성 건수 (목표 0건) |
| **경험/성능** | 응답 지연 | 질의→최종 응답 P95 (목표 ≤ 6초) |
| | 종합 응답률 | 주차+교통을 모두 포함해 답한 비율 |

- 해커톤 데모 합격선: 실제 핫스팟 질의에 **실데이터 기반 주차+교통 종합 추천**이 환각 없이, 갱신시각과 함께 6초 내 생성.

## 5. 기능 요구사항 (요약)
| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F1 | 자연어 목적지 입력 → 핫스팟 매핑 | Must |
| F2 | 지역 실시간 도로소통 조회 (RouteAgent) | Must |
| F3 | 지역 실시간 주차 가용 조회 (ParkingAgent) | Must |
| F4 | 주차+교통 종합 추천 (Coordinator) | Must |
| F5 | 챗 인터페이스(스트리밍) | Must |
| F6 | 매핑 실패 시 후보 제시 / 갱신시각 표기 | Should |
| F7 | 즐겨찾기 목적지 | Could |

## 6. 비목표
- turn-by-turn 정밀 내비, 좌표 기반 경로탐색, 예약/결제, 서울 외 지역, 사용자 위치 추적.

## 7. 데이터 소스
- 서울 실시간 도시데이터(citydata, OA-21285): 핫스팟 단위 `PRK_STTS`(주차) + `ROAD_TRAFFIC_STTS`(도로소통). 인증키 발급 필요.

## 8. 리스크/가정
- citydata 필드명은 매뉴얼로 확정 필요(코드는 방어적 파싱). 핫스팟 외 주소는 근사. LLM 비용은 해커톤 가이드(~$5) 내.
